## Weight Download and Conversion

<a id="jump2.1"></a>

### 1. Weight Download

Download the corresponding model weights from the Hugging Face library:

- [Qwen2-VL-2B](https://huggingface.co/Qwen/Qwen2-VL-2B-Instruct/tree/main)

- [VideoAlign](https://huggingface.co/KwaiVGI/VideoReward/tree/main)

 Save the downloaded model weights to the corresponding directories:

 - Qwen2-VL-2B: `ckpt/hf_path/Qwen2-VL-2B-Instruct`
 - VideoAlign: `ckpt/hf_path/VideoReward`

<a id="jump2.2"></a>

### 2. Weight Conversion (hf2mm)

MindSpeed MM modifies the structure names of some original networks. Use the `mm-convert` tool to convert the original pre-trained weights. This tool implements the conversion from HuggingFace weights to MindSpeed-MM weights. Refer to [Weight Conversion Tool Usage](../../docs/en/features/mm_convert.md).

| Parameter | Meaning and Usage |
| --- | --- |
| mm_dir | Directory to save the converted weights |
| hf_dir | HuggingFace weight directory |
| pt_path | Path to the VideoAlign model in `.pth` format |
| llm_pp_layers | Number of LLM layers partitioned across each card. Note that this must be consistent with `pipeline_num_layers` configured in `model.json`. |
| vit_pp_layers | Number of ViT layers partitioned across each card. Note that this must be consistent with `pipeline_num_layers` configured in `model.json`. |
| tp_size | Number of TP partitions. Note that this must be consistent with the configuration in the fine-tuning startup script. |
| resize_vocab_size | `vocab_size` after applying evaluation metrics to tokens<br>The `embed_tokens.weight` layer of the Qwen2-VL-2B model will be resized according to the `vocab_size` change.<br>The corresponding `vocab_size` for ['VQ', 'MQ', 'TA'] scoring metrics is `151660`. |
| model_prefix | Remove the prefix (`base_model.model.`) generated by the PEFT wrapping in the VideoAlign weights from HuggingFace. |
| new_transformers_weight_key | Whether to use the new Qwen2VL weight naming convention for HuggingFace weights.<br>If the HuggingFace weights use the new Transformer weight names: `model.language_model.layers.xx` and `model.visual.blocks.xx` (original weight names: `model.layers.xx` and `visual.blocks.xx`), set the following command:<br>`--cfg.common_model_config.new_transformers_weight_key true \` |
| enable_canonical_hf_struct | Whether the MM weights use the standard HuggingFace model structure. `true`: Use the HuggingFace transformers model structure; `false`: Use the native Megatron model structure. The default value is false. It is recommended to enable this when using LoRA fine-tuning. |

```bash
# Convert Qwen2VL weights to MM format for training.
mm-convert  VideoAlignConverter hf_to_mm \
  --cfg.mm_dir "ckpt/mm_path/Qwen2-VL-2B-Instruct" \
  --cfg.hf_config.hf_dir "ckpt/hf_path/Qwen2-VL-2B-Instruct" \
  --cfg.parallel_config.llm_pp_layers [[28]] \
  --cfg.parallel_config.vit_pp_layers [[32]] \
  --cfg.parallel_config.tp_size 1 \
  --cfg.common_model_config.resize_vocab_size 151660 \
  --cfg.common_model_config.enable_canonical_hf_struct true

# Convert VideoAlign weights to MM format for fine-tuning/inference/evaluation.
mm-convert  VideoAlignConverter hf_to_mm \
  --cfg.mm_dir "ckpt/mm_path/VideoReward" \
  --cfg.hf_config.hf_dir "ckpt/hf_path/Qwen2-VL-2B-Instruct" \
  --cfg.pt_path "ckpt/hf_path/VideoReward/checkpoint-xxx/model.pth" \
  --cfg.parallel_config.llm_pp_layers [[28]] \
  --cfg.parallel_config.vit_pp_layers [[32]] \
  --cfg.parallel_config.tp_size 1 \
  --cfg.common_model_config.model_prefix "base_model.model." \
  --cfg.common_model_config.enable_canonical_hf_struct true

```

If you need to train with the converted model, synchronously modify the `LOAD_PATH` parameter in `examples/videoalign/finetune_lora.sh`. This path points to the converted or sharded weights. Note that it should be distinguished from the original weight path.

```shell
LOAD_PATH="ckpt/mm_path/Qwen2-VL-2B-Instruct"
```

Or

```shell
LOAD_PATH="ckpt/mm_path/VideoReward"
```

### 2. Parameter Configuration

[Data Directory Configuration]

Modify the dataset paths in `data.json` according to the actual situation, including fields such as `model_name_or_path`, `data_folder`, `data_path`, and `cache_dir`. Note that the weight path for `model_name_or_path` is the path before weight conversion.

**Note: Do not configure the same mounted directory for `cache_dir` across multiple machines to avoid conflicts caused by writing to the same file.**

```json
{
    "dataset_param": {
        "dataset_type": "rewardvideo",
        "preprocess_parameters": {
            "model_name_or_path": "./ckpt/hf_path/Qwen2-VL-2B-Instruct/",
            ...
        },
        "basic_parameters": {
            ...
            "data_folder": "./datafolder/VideoGen-RewardBench/",
            "data_path": "./datafolder/VideoGen-RewardBench/videogen-rewardbench.csv",
            "cache_dir": "./cache_dir",
            ...
        },
        ...
    },
    ...
}
```

[Model Directory Configuration]

Modify the training configuration in `model.json` according to the actual situation, including:

| Parameter | Meaning and Usage |
| ---  |  ---|
|`output_dim`| Model output dimension. |
|`reward_token`| Calculation method for reward_token. |
|`loss_type`| Calculation type for reward loss. |
|`lora_mixed_training`| Enable mixed training of LoRA and non-LoRA parameters (e.g., full-parameter fine-tuning for ViT; LoRA fine-tuning for text-decoder). |
|`lora_apply_modules`| Modules using LoRA fine-tuning during mixed training. |
|`lora_save_full_weight`| Save both LoRA weights and non-LoRA weights during mixed training. |
|`freeze`| Freeze the parameters of this module. Excluding it from training. |

```json
{
    "model_id": "qwen2vl",
    "img_context_token_id": 151656,
    "video_token_id": 151656,
    "vision_start_token_id": 151652,
    "output_dim": 1,
    "reward_token": "special",
    "loss_type": "btt",
    "lora_apply_modules": ["text_decoder"],
    "lora_mixed_training": true,
    "lora_save_full_weight": true,
    ...
```

```json
"image_encoder": {
        "vision_encoder": {
            ...
            "freeze": true,
            ...
```

Note: The default configuration uses full-parameter training for the image-encoder and LoRA fine-tuning for the text-decoder in a mixed training approach; otherwise, insufficient memory is likely to occur.

[ViT Module Recompute Configuration (Optional)]

TO use ViT training (ViT is frozen in the default configuration; to unfreeze it, configure the `"vision_encoder": {"freeze": false}` in the `vision_encoder` part in the `model.json` file), you can enable recomputation to reduce memory usage (note that this will impact performance).

To enable ViT recomputation, add recomputation-related parameters to the `vision_encoder` part in `model.json`. The `recompute_granularity` parameter allows you to configure the recomputation granularity as `full` or `selective`.

1. `full` mode

    All components within the TransformerLayer (layernorm, attention, mlp) are recomputed, and the number of layers to recompute can be configured.

    - `recompute_method`: Controls how the number of recomputation layers is calculated. Options are `uniform` (uniform recomputation) or `block` (block-wise recomputation).
    - `recompute_num_layers`: Controls the number of layers to recompute.

    Example configuration:

    ```json
    {
      "model_id": "videoalign",
      "img_context_token_id": 151656,
      "video_token_id": 151656,
      "vision_start_token_id": 151652,
      ...
      "image_encoder": {
        "vision_encoder": {
          ...
          "recompute_granularity": "full",
          "recompute_method": "uniform",
          "recompute_num_layers": 1
        }
      },
      ...
    }
    ```

2. `selective` mode

    Only recomputes the core_attention component of the attention in TransformerLayer. This mode is not available in LoRA scenarios.

    Example configuration:

    ```json
    {
      "model_id": "videoalign",
      "img_context_token_id": 151656,
      "video_token_id": 151656,
      "vision_start_token_id": 151652,
      ...
      "image_encoder": {
        "vision_encoder": {
          ...
          "recompute_granularity": "selective"
        }
      },
      ...
    }
    ```

[Model Saving, Loading, and Logging Configuration]

Configure the parameters of `examples/videoalign/finetune_lora.sh` according to your actual situation, including load and save paths, and the save interval `--save-interval` (Note: Distributed optimizer files are large and saving them takes a long time, so please set the save interval carefully).

```shell
...
# Load Path
LOAD_PATH="ckpt/mm_path/Qwen2-VL-2B-Instruct"
# Save Path
SAVE_PATH="save_dir"
...
GPT_ARGS="
    ...
    --no-load-optim \  # Do not load optimizer state. Remove this if loading is required.
    --no-load-rng \  # Do not load random number state. Remove this if loading is required.
    --no-save-optim \  # Do not save optimizer state. Remove this if saving is required.
    --no-save-rng \  # Do not save random number state. Remove this if saving is required.
    ...
"
...
OUTPUT_ARGS="
    --log-interval 1 \  # Logging Interval
    --save-interval 5000 \  # Save Interval
    ...
    --log-tps \  # Enabling this parameter prints the average sequence length of the language module at each step during training, and calculates the throughput in tokens per second after training completes.
"
```

To load weights, optimizer states, and other components from a specific iteration, set `LOAD_PATH` to `"save_dir"` and modify the `latest_checkpointed_iteration.txt` file to specify the desired iteration count (not supported now).

```shell
$save_dir
  ├── latest_checkpointed_iteration.txt
  ├── ...
```

If LoRA mixed training is enabled, the saved weights include both LoRA and non-LoRA weights, which must be sharded using a conversion script before loading separately (not supported now).

[Single-Node Configuration]

Configure the parameters in `examples/videoalign/finetune_lora.sh` as follows:

```shell
# Modify the ascend-toolkit path according to your actual setup.
source /usr/local/Ascend/cann/set_env.sh
NPUS_PER_NODE=8
MASTER_ADDR=localhost
MASTER_PORT=6000
NNODES=1
NODE_RANK=0
WORLD_SIZE=$(($NPUS_PER_NODE * $NNODES))
```

Note: FA is enabled by default during training; otherwise, insufficient memory errors may occur. This is controlled via the `--use-flash-attn` switch.

Note that when PP is enabled, `pipeline_num_layer` configured for `vision_encoder` and `text_decoder` in `model.json` control their respective PP partitioning strategies. For pipeline parallelism, `vision_encoder` must be processed before `text_decoder`.
For example, the default values for the 7B model are `[32,0,0,0]` and `[1,10,10,7]`. This means that within the PP domain, the 32 layers of `vision_encoder`, followed by 1 layer of `text_decoder`, are placed on the first card; the next 10 layers of `text_decoder` are placed on the second card; the next 10 layers of `text_decoder` are placed on the third card; and the next 7 layers of `text_decoder` are placed on the fourth card. The layers of `text_decoder` cannot be placed before layers of `vision_encoder` are fully placed (for example, the configuration, `[30,2,0,0]` and `[1,10,10,7]`, is incorrect).

Also note that if all parameters on a certain card are frozen, it will result in no gradients (for example, when `vision_encoder` is frozen with `[30,2,0,0]` and `[0,11,10,7]` configured for PP). In this case, you need to add `--enable-dummy-optimizer` to the `GPT_ARGS` parameter in `finetune_qwen2vl_7b.sh`. For more details, refer to the [dummy_optimizer](../../docs/en/features/dummy_optimizer.md).

## Notes

1. In `finetune_lora.sh`, parameters related to the model structure do not take effect; the configuration of parameters with the same name in `examples/videoalign/model.json` takes precedence. Training-related parameters that are not part of the model structure should be modified in `finetune_lora.sh`.
